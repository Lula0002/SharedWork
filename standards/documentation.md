# Documentation Standards

## Markdown files (this repo)

- One idea per file. Short is better than long.
- Use tables for comparisons, bullet points for lists, headers to navigate.
- Write in present tense: "Authors squash merge" not "Authors should squash merge."

## Code docstrings

- Public functions and classes get a one-line docstring describing what they do.
- Skip docstrings for private helpers where the name is self-explanatory.
- No multi-paragraph docstrings — if you need that much explanation, the code needs simplifying.
